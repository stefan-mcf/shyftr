# ShyftR Naming Conventions

> Converted from PDF to Markdown for repository reference. Source PDF pages: 4.

This document records the current naming recommendations for the major concepts in ShyftR. The goal of this scheme is to use terms that are memorable, easy to teach and understand, and consistent with the project’s "battery/memory cell" theme. Where previous implementations used different names, a mapping is provided to aid migration.

## Guiding Principles

- Clarity over cleverness – Names should immediately convey their role in the lifecycle. Readers

should not need to understand the entire system before they can reason about a given term.

- One-to-one mapping – Each stage of the memory lifecycle should have exactly one canonical

name. Avoid synonyms that could cause confusion.

- Theme coherence – The electrical storage metaphor (cells, charges, circuits) is used consistently

to hint at how information moves through the system.

## Canonical Terms

The following table lists the official vocabulary with a brief definition of each. Later sections provide more detailed explanations. Concept Definition Cell A bounded memory store. Each cell encapsulates its own ledger files and indexes and can be mounted on an agent, a project, a user or a sector. Pulse Raw evidence entering the system, such as logs, task closeouts or documents. Pulses are captured verbatim in append-only ledgers. Spark A candidate memory extracted from a pulse. Sparks are proposed statements or heuristics that require review before being promoted. Charge An approved, durable memory. Charges are the authoritative facts, rules or heuristics used to build packs for tasks. Circuit A distilled pattern or theme derived from multiple related charges. Circuits represent reusable workflows or multi-step heuristics. Rail A high-authority rule or doctrine promoted from repeated circuits. Rails are shared across cells and act as guardrails or policies. Grid The retrieval layer responsible for searching charges (e.g. sparse FTS, vector index). Grids are rebuildable and never the source of truth. Pack A bounded, task-ready context compiled from charges and circuits. Packs include guidance, caution items, verification heuristics and conflicts. Concept Definition Signal Outcome feedback after a pack is used. Signals record whether retrieved memory was useful, harmful, ignored or missing. Regulator The policy layer that enforces boundaries on ingestion, review and promotion. The regulator defines what may be automatically promoted or requires human review.

### Detailed Explanations

Cell A cell is the fundamental container for all memory operations. It holds append-only ledger files (pulses.jsonl, sparks.jsonl, charges/approved.jsonl, etc.) as well as indexes under grid/. A cell can be mounted onto any layer of an agent system – such as a user profile, project, runtime sector or domain – allowing the same code to operate in different contexts. Multiple cells can be connected later through federation, but the cell remains the unit of authority and isolation. Pulse (raw evidence) A pulse captures raw evidence that might be important later. It could be a task closeout, a log line, a document, a chat message or a human note. Pulses are stored verbatim in append-only JSONL files and are never modified. Pulses do not influence retrieval directly – they must first be analysed to produce sparks. Spark (candidate memory) A spark is a potential piece of knowledge extracted from a pulse. Sparks include the proposed statement or heuristic and metadata such as source hash, tags and a suggested kind (e.g. guidance, caution, verification). Sparks live in a pending state until they are explicitly reviewed. The review can approve the spark, reject it, split or merge it into multiple sparks, or leave it pending. Charge (approved memory) A charge is what happens when a spark is approved. Charges are durable and authoritative. They are stored in charges/approved.jsonl and become candidates for retrieval. Charges have confidence scores that evolve over time via signals and sweep proposals, and they include provenance back to the original pulse and review events. Circuit (distilled pattern) A circuit represents a recurring pattern, workflow or multi-step heuristic distilled from several charges. For example, multiple charges about testing before deploying might form a single circuit representing the “test-then-deploy” process. Circuits are optional in the MVP, but the term is reserved for this distilled intermediate layer. (Previously this concept was called a “Coil”; it was renamed because “circuit” more clearly conveys connected flow and reuse.) Rail (promoted doctrine) A rail is a high-authority rule or policy promoted from circuits. Rails act as guardrails across cells and define global constraints that should be followed by agents or runtimes. Because rails can affect many cells, they require strict review and careful promotion. Grid (retrieval layer) The grid is responsible for efficient retrieval of charges. Different grid implementations might include sparse full-text search, vector embeddings, hybrid scoring or other indexing strategies. The grid never stores authoritative data – it can always be rebuilt from the canonical ledgers. Grid status includes metadata such as the embedding model, ledger offsets and whether the index is stale. Pack (task-ready context) A pack is the context assembled for a specific task. It contains selected charges, circuits and rails relevant to the query, organised into sections such as guidance, caution items, verification heuristics and conflict warnings. Packs include provenance and scoring information so that agents can trace why a particular charge was selected or suppressed. Packs are bounded to meet token budgets and can be generated in different modes (e.g. conservative, balanced, exploratory). Signal (outcome feedback) A signal records what happened after a pack was used. Signals capture whether retrieved memory was useful, harmful, ignored, contradicted, over-retrieved or missing. This feedback is appended into outcome ledgers and drives learning through confidence events and retrieval affinity adjustments. Signals are essential for making the memory self-learning and adaptive. Regulator (policy layer) The regulator enforces boundaries at every stage of the lifecycle. It determines what pulses are allowed to be ingested, what sparks are acceptable to extract, when a spark can be auto-promoted to a charge, and how retrieval and promotion policies adapt based on feedback. The regulator enables safe automation by preventing unreviewed content from silently becoming durable memory.

## Rationale for Renaming “Coil” to “Circuit”

The prototype codebase uses the term Alloy or Coil to refer to an intermediate layer between charges and rails. However, the name “Coil” proved opaque – users often could not tell what it meant. The new term Circuit retains the electrical metaphor while better conveying that it is a pattern of connected charges that can be reused. A circuit implies a flow or loop of energy, just as a memory circuit represents a recurring pattern of knowledge.

## Summary

The above conventions strike a balance between creativity and clarity. By adopting a clear, themed vocabulary, ShyftR can communicate its concepts to developers, operators and researchers more effectively. The canonical sequence of the memory lifecycle is now: Pulse → Spark → Charge → Circuit → Rail ↓ Pack → Signal Where the regulator governs every transition and the grid facilitates retrieval without becoming an authority. These names make the project’s architecture easier to explain and implement, and they provide a stable foundation for future documentation and API design.
